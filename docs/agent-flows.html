Three agent paths: vmllm · rig · vmagent

All three run the model in the browser. They differ in how the task gets in, how much orchestration wraps the model, and how much of it crosses the serial line back into the VM.

Solid — stays on one side
Dashed — crosses the serial UART

Path 01 · raw model vmllm one chat call — no tools, no loop

The guest fires the request itself and blocks reading the reply inline; the answer streams all the way back over the serial line to the console. One round-trip carries the whole exchange.

Path 02 · lean agent rig browser loop · 4 tools · ≤6 turns

The guest command writes AGENT_RIG straight to the serial and returns immediately (fire-and-forget). The browser drives a minimal tool loop; the answer is painted into the terminal, not sent back over serial.

Path 03 · full agent vmagent Deep Agents loop · planning · full tool suite

The guest stages the request as a file; the browser drains it with a periodic vmagent-poll. From there it mirrors rig but with the heavier Deep Agents orchestrator and its browser-reaching tools.

Transport notes
  • Frames ride a hidden serial line — inittab runs autologin on ttyS0 and autologin-rpc on ttyS1; the browser filters __V86RPC__ markers out of the visible terminal.
  • Bytes are paced ~1/ms to stay under the emulated 115200-baud UART's ~11.5 KB/s ceiling; long replies use ACK'd reliableReply chunks to avoid overrun drops.
  • For rig and vmagent, non-YOLO runs gate write/shell behind a browser confirm() before the tool crosses into the VM.
